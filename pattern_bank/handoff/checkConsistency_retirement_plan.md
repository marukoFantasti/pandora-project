# checkConsistency() 棚卸し設計書 — パターンバンク方式導入に伴う縮退計画

作成日: 2026-07-14
前提資産: `patterns_g02.json` (v0.6, 21パターン) / `generate_poc_v06.py` (決定論ジェネレータ, 33,000問ストレステスト欠陥ゼロ)

## 0. 結論サマリー

- **パターン生成経路では、ランタイムの checkConsistency() 呼び出しを全廃できる**（Haiku呼び出しゼロ）
- 11基準の大半は、生成後の「検出」から生成前の「構造保証」に移る
- checkConsistency() 本体は**削除せず温存**する。適用対象は「AI自由生成が残る経路」のみに限定
- `findOutOfGradeKanji()`（機械照合）は共有ユーティリティとして昇格し、**テンプレート登録時チェック**に転用する

## 2. 経路別のチェック構成（移行後）

### 経路A: パターン生成（新設・小2文章題）
```
パターン選択 → 数値サンプリング(制約充足) → テンプレート充填 → 完成
  ランタイムAI呼び出し: 0回
  ランタイムチェック: verify()の3項目(漢字/数値由来/正値) ※保険としての残置。理論上は常時PASS
```
verify() は残すが、FAILは「ユーザーに再生成を促す」ではなく**「バンク設計バグとしてコンソールにエラー記録」**とする。33,000問テストで欠陥ゼロなので、発火＝バンクかジェネレータの実装バグを意味する。

### 経路B: AI自由生成（既存・当面残る領域）
- 対象: 時計読み取り(c02)・アレイ図(c47)等の図依存問題、パターン未整備の学年(g01, g03〜g06, jhs)、英語・国語系
- checkConsistency() 11基準を**現状のままフル適用**
- パターン整備が学年ごとに進むたび、その学年の算数文章題を経路Aへ移す

### 経路C: テンプレート登録（新設・オフライン工程）
- 新パターン/新テーマのテンプレート追加時に1回だけ実行
- チェック内容: ①findOutOfGradeKanji（機械）②制約充足可能性テスト（ジェネレータの棄却例外で検出）③ストレステスト1000問 ④日本語自然さ（AI検品→人間最終検品）
- ここを通ったテンプレートは、経路Aで無審査生成してよい

## 3. コード変更方針（Claude Code向け）

1. `checkConsistency()` は**変更しない**（経路Bで現役続行）
2. `findOutOfGradeKanji()` を共有モジュールに切り出し、登録時ツール（経路C）から呼べるようにする
3. `pandora_main.html` に生成モード分岐を追加: unit_id が patterns_g02.json にパターンを持つ場合はパターン生成、無ければ従来のAI生成＋checkConsistency()
4. パターン生成のログは `[patternGen]` プレフィックスで記録（checkConsistencyの `[checkConsistency]` と区別）

## 5. リスクと保険

- **リスク**: 本設計書の基準1〜8のマッピングは記録からの再構成であり、実コードに未知の基準が含まれる可能性がある → §4の突き合わせで解消
- **保険**: 経路Aにも verify() 3項目を残置（実行コスト無視できる程度）。エラー発火時はバンクのバグとして扱う
- **不変条件**: 経路Bの品質は本移行で一切変化しない（checkConsistencyは無変更のため）

## 6. タスク0突き合わせ結果による追記

タスク0（実コードの基準1〜11文言とのマッピング確認）で判明した内容を反映する。

### 6.1 経路Cチェックリストへの追加

§2「経路C: テンプレート登録」のチェック内容①〜④に加え、以下を追加する：

- **⑤テンプレート末尾が問いの形か**：正規表現lintで機械チェックする。`sentence_templates`の各文字列が「ですか。」「ますか。」「なさい。」等の疑問文・指示文の語尾で終わっているかを検証し、該当しないテンプレートは登録時に拒否する（基準4対応）。
- **⑥`sentence_templates`のプレースホルダがホワイトリスト内か**：`{...}`形式のプレースホルダが、そのパターンの`slots`・`quantity_slots`の`_disp`表示用スロット・宣言済み語彙スロットのみで構成されているかを機械チェックする。`{ans}`や`computed_slots`（本来`answer_template`側でのみ使うべき計算結果）を`sentence_templates`（本文側）に書いてしまっている場合は登録時に拒否する（基準5対応：本文に答えを漏らさないことの機械的保証）。

### 6.2 基準6（level別の式表示可否）の整理

パターン生成経路では、`sentence_templates`（本文）に計算式・矢印図・中間結果を含めない設計を維持する限り、confirmation/practiceレベル相当の要件は**構造保証**される（テンプレートに式を書かなければ、式が表示されようがない）。example_follow（例題）で許可されている「式を生徒に見せる」という役割は、生成ロジック側で式を本文に混ぜ込むのではなく、**`answer_template`を問題文とは別に並記表示する**（表示側の対応）ことで実現する。したがって基準6は、パターン生成経路そのものには介入せず、テンプレート作成規約（本文に式を書かない）＋表示側の設計方針として扱う。

### 6.3 経路Cの適用対象の明記

経路C（①〜⑥のチェック一式）は、**「新パターン追加時」と「新テーマ追加時」の両方**で全項目を実施する。テーマ追加はテンプレート文言そのものを差し替える（`themed_templates`）ため、新パターン登録時と同じリスク（語彙・プレースホルダ・末尾表現の逸脱）を持つ。
